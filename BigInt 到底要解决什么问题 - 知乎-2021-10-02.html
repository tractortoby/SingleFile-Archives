BigInt 到底要解决什么问题

A-yon
Web 爱好者，GitHub 与知乎专栏: hyurl

前几天更新了一下 VSCode，然后发现在输入 typeof 的时候，提示器多出来了一个新的类型 bigint。头脑中第一个念头是：这是什么鬼？然后秉持乐于学习的态度，上 MDN 翻了一下资料。

BigInt is a built-in object that provides a way to represent whole numbers larger than 2^53, which is the largest number JavaScript can reliably represent with theNumber primitive. （BigInt 是一个内置的对象，它提供了了一种方式来表示所有大于 2^53 的数字，也就是 JavaScript 原始类型 Number 能够能够可靠表示的最大数字。）

Ba la... ba la... ba la...

介绍了一大堆这个类型怎么创建，如何用字面量表示，如何比较和计算等等。唯独不见兼容性测试列表。没办法，又只能继续 Google，终于在 GitHub 上找到了蛛丝马迹。

这个类型目前还只是提案阶段，虽然浏览器（大概是 Chromium 67+）已经提供了支持。

提案上对 BigInt 的描述是：

JavaScript 中的任意精度整数。

这个有点意思，以往经常说浮点数有精度问题，这个整数的精度又是什么？BigInt 到底又要解决什么问题？几天来一直没有时间去做详细了解，脑中的迷雾就这么一直留着。直到今天在群里和人讨论才想到 Number.MAX_SAFE_INTEGER 和 JS 中科学计数法的问题。

一、长整数表示

以往，当我们输入一个比较长的数字的时候，JS 总是将其转换为科学计数，就是那个加了 e 的模样，即使调用 toString() 方法，也无法获得它表示的真实数字。但我用 BigInt 字面量尝试了一下，发现它并不存在这个问题了，而是能够正确地显示输入的数字。例如

1111111111111111111111111111111111111 // => 1.1111111111111112e+36
1111111111111111111111111111111111111n // => 1111111111111111111111111111111111111n


而调用 toString() 方法，依旧能够获得期望的数值。

二、精确计算

另一个在编程语言中比较著名的问题，就是数学计算的精度问题。人们通常认为，计算机出现的初衷，就是帮助人们计算复杂的数学，那么按理说应该在数学计算上，计算机应该非常准确才对。但实际上结果却令人大跌眼镜。

例如在 JavaScript 中，语言所能够保证“安全”（精确）地进行计算的最大整数是 2^53 - 1 ，也就是 9007199254740991 （常量 Number.MAX_SAFE_INTEGER）超过了这个数值，JS 就会产生一些意想不到的问题。比如：

9007199254740991 + 1 // => 结果是 9007199254740992
9007199254740991 + 2 // => 结果也是 9007199254740992

我想对于绝大多数人（包括我自己）来说，都很吃惊。由于设计上的天然缺陷，语言无法通过计算机硬件来直接进行大数字计算。为了解决这个问题，JavaScript 必须要引入第三方库，例如 Math.js，才能用于数学计算。

而现在，JavaScript 语言终于要迈出内建支持的第一步，提出了 BigInt 的概念。例如下面的计算就不再是问题：

9007199254740991n + 1n // => 结果是 9007199254740992n
9007199254740991n + 2n // => 结果是 9007199254740993n


当然这只是一小步，BigInt 解决的只是整数计算的精度问题，更著名的浮点数精度问题依旧没能解决，可能以后又会有 BigDecimal，BigNumber 之类的概念提出来（PS: BigInt 也支持其它进制）。

只是不知道以后字面量和原始类型又该如何表示了，毕竟 BigInt 的原始类型已经写死了 bigint （typeof 返回 bigint），假如以后又加入一个 BigNumber 支持小数，即使字面量依旧可以用 123.123n 这样的形式，那么 typeof 又该返回什么值呢？
