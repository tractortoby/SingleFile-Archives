首发于分布式与机器学习

为什么要读《Designing Data-Intensive Applications》

HaoC
​
各种好玩的东西

TL;DR

一句话概括，本书讲的是，设计数据应用时，要考虑和遵循的各种分布式设计原理/原则。


本书侧重的是数据，设计分布式系统要解决的首要问题，即数据的存储、访问、一致性、可用性等等，不过在最后一部分也讲了数据处理模型。

这本书讲的是原理而不是各种数据库 API 、SQL 应该怎么用。当你理解了各种设计背后的原理之后，你用起各种数据库来才基本上是得心应手，而不是迷失在 API 的细节里（当然只看本书是远远不够的）。


好在哪里


如果你正在学习分布式系统，推荐读读《Designing Data-Intensive Applications》，中文名 《设计数据密集型应用》，豆瓣评分原版9.6，影印版更夸张 10.0 （截止目前20人评分），清一色好评，真正的好书，深入浅出，达到了没经验的人可以轻松看懂，有经验的人收获更多的目标。如果你学过一门数据库相关的课程，会发现之前学习的各个点终于能够自然的串联起来，这是一种很愉悦的感觉。

作者在讲一个概念不只是单纯罗列，更会讲到具体的工业界实现，经常就是 XX 数据库就是这么实现的，而 XX 数据库是这样实现的，会不会有什么问题。

这就解决了一个很大的问题，平常我们学了一个概念之后，只知道这个概念很有用，但却不一定知道这个概念多有用，现在有没有数据库在用，最佳实践是什么，有没有更好的，要注意的坑是哪些？真的，至少要知道这些，才能真正将一个知识落地。


简单的问题可能是复杂的

例如第8章就讲到了时钟问题，Monotonic(单调递增的时钟，不会被 NTP程序调整) 和 Time-of-Day（实际时间，例如UTC标准时间） Clock。事实上，计算机的时钟并不是绝对精确的，有时快有时慢，需要通过 NTP 和时间服务器进行同步，同步的话就可能涉及到时钟调前调后的问题，对多数业务问题不大，但对于某些业务就是不可接受。比如我需要知道开机到现在运行了多少时间，只能用单调时间而不能用当前的时间减开机时间（可以想想为什么，很简单）。

作者讲了很细节的部分（有种信手拈来之感……），某些 Windows 时间同步会快个10ms的倍数，新的版本问题小一些（less of a problem），这种知识真的只能是经过大量实践的人才可能知道。真实世界永远不可能像论文里面的计算模型，有那么完美的运行环境。

并且提到了， NTP 虽然不会调整 Monotonic Clock 的大小，但是可能会调整它的快慢（类似于石英钟 振动多少次算一秒）。


而到了真正需要考虑 corner case 的时候，作者又事无巨细的将各种你想得到想不到的要点列出来。所以如果你想要实现一个数据库，也可以对照着相应的章节，看看有没有实现对。


具体一点


过去几十年，工业界的数据库有了各种有趣的发展，从 SQL 到 NoSQL，从单机到分布式、大数据、MapReduce、云服务，从磁盘到SSD、内存……都是为了适应需求的增长：

  • 大公司每天收集的数据量可以达到 TB 甚至PB 级别，需要处理、分发的数据量就更大了（MySQL、PostgreSQL、HBase等等，传统数据库并没有那么弱，MySQL在 Facebook、BAT仍然被大量使用）
  • 业务需要敏捷、响应要快，不能每次用户一个访问都需要从海量数据库查找（路由、分区、缓存等等，Redis、Memcached、LevelDB等）
  • 屏蔽复杂性，对于小团队、小公司、创业公司更友好的分布式设施（云服务）
  • 对于金融等敏感业务数据需要更高的可用性、事务性、容灾备份等等

本书主要分为三个部分：

  • 第一部分讲数据库基本概念，例如 ACID、数据模型、数据存储等等，并且讲了具体的实现（各种图表、形式化描述、甚至还有 shell）
  • 第二部分是从单机到分布式，怎么去处理可扩展性、一致性的问题
  • 第三部分是数据处理，批处理（MapReduce）、流处理（Flink、Storm）、数据集成分析等


怎么读


里面的英文真的很简单，大部分的术语平常也接触得到，相信大部分程序员都可以轻松上手。当然看英文版确实会慢一点，不过这类书不是小说，适当慢一点反而有好处呢！

可以快速的读一遍，书的内容基本上是 self-contained 的，如果你上过数据库方面的课程那么基本上基础就够了。

书后面有各种引用、文章，读第一遍的时候完全可以略过去，要把它们读完要耗费的可能是读好几本书的精力（可以作为手册查阅）。
